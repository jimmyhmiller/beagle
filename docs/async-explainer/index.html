The mental model

1 · Sequential by default

No callbacks, no async keyword. Code runs top to bottom. A read returns the bytes; the next line uses them.

2 · Non-blocking, not blocking

“Sequential” ≠ “blocking”. When a flow hits sleep / a socket read, its continuation is parked so other flows run on the same thread.

3 · future to go concurrent

To break out of program order within a flow, wrap work in spawn/future and await it later.

How it actually works

The runtime installs a cooperative scheduler as the ambient async handler for main() and every thread(). An async op (sleep, socket/file IO) is an algebraic effect: the handler captures the continuation and parks it instead of blocking the OS thread, then the scheduler resumes it when the IO completes. Many logical flows (e.g. many client connections) interleave on one thread — each flow still reads as plain sequential code.

one OS thread, cooperative scheduler ┌──────────────────────────────────────────────────────────┐ │ accept-loop ──park(accept)──┐ │ │ conn A handler ─park(read)─┐│ event loop (epoll/kqueue) │ │ conn B handler ─park(read)┐││ ▲ │ │ │ ↓↓↓ │ ▼ ready │ │ scheduler: resume whichever IO completed first │ └──────────────────────────────────────────────────────────┘ sequential INSIDE a flow · concurrent ACROSS flows

There are three handlers. Cooperative (default) parks continuations on one thread. Blocking (opt-in, deterministic tests) runs everything inline. The spawn/future path uses real OS threads for genuine parallelism. The scenarios below prove each behavior.

Scenarios — run them

Scenario 1 · sequential by default

Plain top-to-bottom code

No futures, no callbacks. Three steps run in order — this is what “you don’t have to think about async” means.


    
Expect: A, B, C strictly in order.

  


  
  

    
Scenario 2 · future = opt-in concurrency

      
The same two sleeps, sequential vs. wrapped in future

      
Written sequentially, two 200 ms sleeps take ~400 ms. Wrap each in spawn and they overlap to ~200 ms — concurrency is something you ask for.

    

    

    

    
Expect: sequential ≈ 400 ms, future-wrapped ≈ 200 ms.

  


  
  

    
Scenario 3 · many clients, one thread

      
A server whose handler is plain sequential code

      
Each connection handler is a simple loop { read; write }. Two clients are served concurrently on a single thread — client A is parked mid-read while B connects and is served. The thread ids prove it.

    

    

    

    
Expect: both clients echoed; all three thread ids equal.

  


  
  

    
Scenario 4 · error isolation

      
One connection fails; the server keeps serving

      
A handler’s try/catch works even though read/write park the continuation in between. One client triggers a thrown error; the next client is still served. (This is the bug class that used to crash the server — now fixed.)

    

    

    

    
Expect: the error is caught server-side; client 2 gets ok:hello.

  


  
  

    
Scenario 5 · why the default matters

      
Cooperative (default) vs. the blocking handler

      
Run the identical workload under the default cooperative handler and under the explicit run-blocking handler. The blocking one runs spawn’d work inline, so it serializes — proving the default is what gives you overlap.

    

    

    

    
Expect: default ≈ 200 ms, blocking ≈ 400 ms.
