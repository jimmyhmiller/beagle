## The architectural principle (read this first)
**`socket` is the transport + concurrency layer; `http`/`ws` are protocol layers that plug a callback into it.** Do NOT let http/ws reimplement the accept loop or import `beagle.async`.

`socket` already has the concurrency primitive:
```beagle
// beagle.socket.bg
fn on-connection(listener, callback) {
    loop {
        let conn = accept(listener)
        async/spawn(fn() { callback(conn) })   // one cooperative task per connection
    }
}
```
A prior attempt added a manual `while accept { async/spawn(handle) }` loop into `http/serve` and `ws/serve-ws` (plus an `async` import in each). That was REVERTED on purpose — it duplicates `on-connection`. Use `on-connection` instead.

## Tasks, in order

### 1. Concurrent serve — finish cleanly (SMALL, do first)  ✅ DONE
> `http/serve` and `ws/serve-ws` now delegate to `socket/on-connection(listener, fn(conn){...})` (no `async` import). Validated: a `/slow` (`async/sleep(2000)`) handler returns in ~2.03s while a `/fast` one fired 0.2s later returns in ~0.02s — concurrent.

`http/serve` and `ws/serve-ws` currently run `handle-connection` synchronously in their own accept loop → serial (one slow client blocks all). Replace with delegation to `socket/on-connection`:
```beagle
fn serve(host, port, handler) {
    let listener = socket/listen(host, port)
    println("beagle.http serving on ${host}:${port}")
    socket/on-connection(listener, fn(conn) { handle-connection(conn, handler) })
}
```
Same shape for `ws/serve-ws` → `socket/on-connection(listener, fn(conn) { handle-ws-connection(conn, on-message) })`. No `async` import in http/ws. (The cooperative scheduler reaps completed tasks; `handle-connection` already wraps errors in try/catch.)
**Validate:** a `/slow` handler (`async/sleep(2000)`) must not block a `/fast` one — start a server in the background, hit both, confirm fast returns immediately.

### 2. Unified socket read layer (MEDIUM — the keystone)  ✅ DONE
> Lifted `byte-length`/`byte-slice` DOWN into `beagle.string-builder` (`sb/byte-length`/`sb/byte-slice`). Added `socket/reader(conn)` + `socket/reader-with(conn, pull)` → a `Reader` with `read-line`/`read-n`/`read-n-buf` (byte-faithful builder for ws frame headers)/`read-all`. Transport is a `pull(n)` thunk (raw socket default; TLS via `reader-with(conn, fn(n){tls/read(conn,n)})`; test harness can feed canned bytes). `read-line` is O(n) now: tracks a scan offset AND an incrementally-maintained buffer length (byte-length is only ever computed on the freshly pulled chunk, never the whole buffer). http dropped ConnBuf/cb-fill!/read-line/read-n/read-until-eof/byte-*; ws dropped read-exact-buf and now shares ONE reader across the handshake + frame loop (fixed a latent bug: bytes read past the upgrade request were dropped). Validated: GET, POST Content-Length, chunked, UTF-8 byte-faithful body, Beagle client round-trip.

http and ws each reimplement byte buffering: `ConnBuf`/`cb-fill!`/`read-line`/`read-n`/`read-chunked-body` in `beagle.http.bg`; `read-exact-buf` in `beagle.ws.bg`. Lift a buffered reader to **socket**: `socket/reader(conn)` → `read-line` / `read-n` / `read-all`.
- Fix http's `read-line` while moving it: it rescans from offset 0 on every `cb-fill!` → O(n²) on a long/unterminated line (Slowloris). Track a scan offset.
- **Blocker to design around:** the byte primitives `byte-length` / `byte-slice` live in `beagle.http.bg`, and `string-index-byte` / `append-range!` in `beagle.string-builder`. `socket` is a DEPENDENCY of `http`, so a socket-level reader can't import http. First move `byte-length`/`byte-slice` (and friends) DOWN into `beagle.string-builder` (or a new `beagle.bytes`), then build `socket/reader` on those, then migrate http+ws to it.

### 3. Timeouts (MEDIUM — the real DoS defense)  ⚠️ READ DEADLINE DONE; caps/wiring OPEN
> **Read deadline done, no runtime change needed** — the runtime already had a timer heap (`timer-set`/`timer-take-completed`/`timer-cancel`) and fd cancel (`tcp-close`). Added a cooperative-scheduler RACE primitive: `PendingOp.Race { branches }` + `RaceBranch { op, cancel, win }`; refactored `poll-tasks` to factor `poll-pending-op(op)` (used by both single-op tasks and race branches); first ready branch wins, losers' `cancel` thunks run. New `Async.IOTimeout { action, ms }` submits the read AND a timer and parks on the race; `socket/read-timeout(socket, n, ms)` rides it — on timeout it CLOSES the socket (drops the pending read = the Slowloris disconnect) and throws. Blocking/Implicit handlers throw a clear "requires cooperative scheduler" error (no silent stub). Also fixed `future(expr)` (lowered to a missing `__make_future` → now wired to `spawn`). Tests: `socket_read_timeout_test.bg` (timeout fires + data-arrives wins), `socket_read_timeout_concurrent_test.bg` (5 deadlined reads overlap, ~500ms not ~2500ms). Example: `resources/examples/read_timeout_example.bg`.
>
> **Reader + http/serve deadline DONE (drops, no 408 body).** `socket/reader-with-timeout(conn, ms)` (a Reader whose every fill goes through `read-timeout`); `http/serve-with(host, port, opts, handler)` with `:read-timeout-ms` (default 30000), `serve` delegates to it; `handle-connection(conn, handler, read-timeout-ms)` reads via the timed reader. A stalled/Slowloris request is DROPPED at the deadline (read-timeout closes the socket = reclaim the slot), matching Go's `ReadTimeout`. Test: `http_read_timeout_test.bg` (drives `handle-connection` directly — serve loops forever, can't be joined by a test). **NO 408 BODY:** delivering one is blocked by the runtime event-loop model — once the read deadline fires, cancelling the pending read requires `tcp-close` (the only cancel), so the socket is already closed; an attempt to keep it open + write 408 PARKS FOREVER (a socket with a pending read won't service a new write). Delivering a 408 needs a runtime primitive: cancel-read-without-close, or full-duplex read+write on one socket.
>
> **HTTP size caps DONE (431/413).** `socket/read-line-max(r, max-bytes)` (+ `read-line-limited`) throws `socket/ReadLimitExceeded` when a line exceeds the cap — checked DURING buffering (bounds memory to ~cap + one chunk for an unterminated line) AND after the terminator (rejects a complete-but-oversized line). http: `ServerConfig { read-timeout-ms, max-header-bytes, max-body-bytes }` (`config-from-opts`, defaults 30000 / 16384 / 10485760); `serve-with` opts `:max-header-bytes`/`:max-body-bytes`; `read-request-limited` caps each header/request line (→ `ReadLimitExceeded` → **431**) and rejects a `Content-Length` over the body cap (→ `BodyTooLarge` → **413**, before reading the body). `handle-connection(conn, handler, config)` maps these via `limit-error-response`. Unlike the 408 case these ARE deliverable: no pending read blocks the write. Test: `http_size_caps_test.bg` (431/413/200, incl. an oversized CHUNKED body → 413). Chunked bodies ARE capped now: `read-chunked-body-max(cb, max-line, max-body)` caps each chunk-size line (→431) and rejects a chunk that would push the body over the cap BEFORE allocating (→413).
>
> **WS FRAME CAP DONE (close 1009).** `ws/read-frame-max(r, max-frame)` (read-frame delegates with null) returns a `FrameTooBig { limit, declared }` marker WITHOUT reading the payload when a frame declares more than the cap; `ws/frame-too-big?` tests it; `ws/close-frame-status(code)` encodes a close with a 2-byte big-endian status. `handle-ws-connection(conn, on-message, max-frame)` sends a 1009 close + drops on an oversized frame; `serve-ws`→`serve-ws-with(host, port, opts, on-message)` with `:max-frame-bytes` (default 1048576). Test: `ws_frame_cap_test.bg` — SOCKET-FREE (canned-bytes reader via `socket/reader-with`, the task-6 harness): oversized→FrameTooBig, small→parses, close-frame-status(1009) byte-checked.

> **WS CLIENT FRAMES + CLOSE-CODE DONE.** Client->server frames must be masked (RFC 6455 §5.3): `ws/encode-frame-masked(opcode, payload)` (random 4-byte key) + `encode-frame-masked-with-key(...)` (explicit key, for tests); public `text-frame-masked`/`binary-frame-masked`/`ping-frame-masked`/`pong-frame-masked`/`close-frame-masked`. Incoming close: `ws/close-code(frame)` (2-byte big-endian, 1005 if no payload) + `ws/close-reason(frame)`. The server read path already unmasks (`parse-frame`). Test: `ws_client_frames_test.bg` — SOCKET-FREE (masked round-trip, byte-level masking with explicit key, server unmask via canned reader, close-code 1009/1000/1005).

> **CLIENT Result SURFACE DONE.** `http/request`/`get`/`post`/`put`/`patch`/`delete`/`head` now return a `Result`: `Result.Ok { value: Response }` or `Result.Err { error: Error.* }` — `Error.Timeout` for a read deadline (opts `:timeout` ms, HTTP path), `Error.IO { message }` for connect/transport/malformed (reuses the canonical std `Error` enum — NO new variants, so no exhaustiveness fallout). > **408 BODY + CONNECT-TIMEOUT DONE (runtime).** Added two event-loop ops: `TcpOperation::CancelRead` (builtin `core/tcp-cancel-read(loop, socket_id)`) drops a pending read + its readable registration but KEEPS the socket open; `TcpOperation::CancelConnect` (builtin `core/tcp-cancel-connect(loop, op_id)`) tears down an in-progress connect by op_id (found via `pending_ops`). IOTimeout's read race now cancel-reads (not close) on timeout, so `reader-with-timeout` → `read-deadline` throws Error.Timeout with the socket STILL WRITABLE → `handle-connection` now sends a real **408** then closes (verified end-to-end). `read-timeout` keeps drop-on-deadline (closes in its own catch). IOTimeout also handles `TcpConnect` → `socket/connect-timeout(host, port, ms)` (Error.Timeout on a slow connect, tears down the connecting fd) → http client opt `:connect-timeout`. Tests: `http_read_timeout_test.bg` now asserts 408; `socket_connect_timeout_test.bg` (success/refused/blackhole-bounded via 192.0.2.1 — outcome-robust). Narrow race: if a connect completes exactly at the deadline, CancelConnect is a no-op and that socket+result leak (bounded). HTTPS read/connect timeout still N/A (TLS read isn't async).
>
> **GENERAL `with-timeout(ms, thunk)` DONE — forced cancellation.** The cooperative scheduler now supports per-flow forced cancellation: `SchedulerTask` carries a `token` + `cleanup` thunk; `ReadyTask` carries the `token`; a module atom `__current-cancel-token` is published by `scheduler-loop` before each resume so async ops a flow performs inherit its token (via `stash-task`/`stash-task-cancellable`); `poll-tasks` checks `task-cancelled?` and, if so, runs `cleanup` (e.g. close the socket of a parked read) and DROPS the task. TCP read/write arms set cleanup = close-socket, connect = cancel-connect. `with-timeout` spawns the thunk under a fresh token (`Async.SpawnWithToken`, now cooperative) and parks on `Async.AwaitWithTimeout` = a Race of [thunk-future, timer]; timer-win cancels the token → the thunk's parked op is cleaned up + discarded → `TimeoutResult.TimedOut`. Thunk completion → `Ok{value}`; thunk throw → propagates. Helpers `async/timed-out?`/`timeout-value`. **ADDITIVE: a null token (every existing flow) skips the check, so non-timeout behavior is byte-identical.** Caveat: single-threaded cooperative — a tight CPU loop that never parks can't be interrupted. Test: `with_timeout_test.bg` (fast Ok, stalled-read force-interrupted+bounded, throw propagates). Blocking/Implicit handlers throw "requires cooperative scheduler".

> **KEEP-ALIVE + REDIRECT-FOLLOWING DONE.** Server: `handle-connection` now loops over HTTP/1.1 keep-alive requests on one connection (reusing the same buffered reader so pipelined bytes carry over; the per-read deadline doubles as the idle timeout) via `serve-one-request` + `should-keep-alive?` (HTTP/1.1 persistent unless `Connection: close`; HTTP/1.0 only if explicit keep-alive). `serialize-response` refactored → `serialize-response-conn(resp, conn-value)` emits the right `Connection: close|keep-alive`. Client: `request` now follows 3xx redirects via `request-following`/`request-once` — `:max-redirects` (default 5, 0 disables); 301/302/303 → GET (body dropped via `without-body`/dissoc), 307/308 preserve method+body; `resolve-redirect-url` handles absolute / root-relative / relative-to-dir Locations (uses `url/last-index-of-char`). > **ASYNC TLS DONE (sans-I/O rustls).** Rewrote `src/builtins/tls.rs` from the blocking `StreamOwned` adapter to SANS-I/O: rustls runs on byte buffers only — `beagle._tls/client-create`/`outgoing`/`feed`/`read-plain`/`write-plain`/`is-handshaking`/`close-notify`/`destroy`. `beagle.tls` now owns an ASYNC `beagle.socket` and shuttles ciphertext between it and rustls (handshake/read/write loops over `socket/connect`/`read`/`write`). So every TLS op PARKS like plain TCP → HTTPS is concurrent and `with-timeout`/`socket/connect-timeout` interrupt it. Wired the http client HTTPS path (`request-tls`) to honor `:timeout`/`:connect-timeout` via new `async/with-timeout!` (value-or-throw-Error.Timeout). > **CONNECTION POOLING DONE — `Client` session.** `http/client()` (or `client-with(max-idle)`) returns a `Client` holding a pool of keep-alive connections keyed by origin (scheme://host:port). `client-request`/`client-get`/`client-post` reuse an idle connection when present, else open one; the request is sent `Connection: keep-alive`, and after a definite-length response (Content-Length/chunked, not `Connection: close`) the connection is returned to the pool (`pool-checkin`, capped at max-idle). `pool-checkout`/`checkin` are atomic via `swap!` + a side atom (idempotent under CAS). A reused connection the server idle-closed surfaces as a transport error → retried ONCE on a fresh connection. Redirects + `:timeout`/`:connect-timeout` work. HTTP and HTTPS alike (async TLS pools like plain TCP). `client-close(client)` drops all idle connections. The bare `get`/`request` stay one-shot/`Connection: close`.
